### 데이터 저장 방식
내부 동작: 메시지가 들어오면 Radix Tree에 데이터를 저장

Pub/Sub과 달리 Consumer가 원하는 시점에 데이터를 가져오는 Pull 방식
특정 ID 이후의 메시지만 가져오거나, 아직 읽지 않은 메시지만 가져오는 등 정교한 읽기 가능

### 소비자 그룹
- 분산 처리: 하나의 메시지를 그룹 내의 딱 한 명에게만 배분
- PEL(Pending Entries List): Redis 내부적으로 누가 어떤 메시지를 가져갔고, 아직 ACK을 안했는지 추적하는 리스트

### Streams를 써야할 때
- 메시지 신뢰성이 중요하여 유실되면 안 될 때
- 대량의 데이터를 여러 소비자가 분산해서 처리해야 할 때
- 단순한 큐가 아닌 검색 간으한 로그 데이터베이스가 필요할 때

### Radix tree를 쓰는 이유
- 단순히 데이터를 꺼내고 지우는 것을 넘어, 방대한 로그 데이터 사이에서 특정 시점의 메시지를 빠르게 찾아내고 메모리를 효율적으로 관리하기 위해
### 특징
- Listpack
	- Redis Stream은 key로서 생성 시각을 id로 가짐. 비슷한 시간대에 생성된 메시지들은 ID의 prefix가 겹치는 경우가 많은데, 이 공통 부분을 공유하므로써 메모리를 아낄 수 있음
### 메시지 브로커로서 List가 아닌 Stream을 사용하는 이유
- At-least-Once 보장
	- List는 데이터를 가져오는 순간 Redis 메모리에서 해당 데이터가 즉시 삭제되지만, 소비자가 데이터를 가져간 직후 로직 오류나 네트워크 문제로 종료되면, 데이터가 영구히 유실됨
	- 그에 반해 Stream의 경우 PEL이라는 내부 구조를 통해 Consumer가 데이터를 읽어가도 XACK 명령을 보내기 전까지는 보류 상태로 유지함
