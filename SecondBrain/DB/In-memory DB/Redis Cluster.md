특정 노드에 장애가 발생하더라도 서비스를 지속할 수 있는 고가용성을 제공하기 위해 설계
## 해시 슬롯
- Redis Cluster는 데이터를 분산하기 위해 전체 데이터를 16384개의 논리적인 슬롯으로 나눔
	- 16384개인 이유는?
- 클러스터에 참여하는 마스터 노드들은 이 슬롯을 나누어 가진다.
- key가 저장될 위치는 CRC16(key) (mod 16384) 공식을 통해 결정됨
	- CRC16을 쓰는 이유는?
## 고가용성과 복제
- Cluster 내의 각 마스터 노드는 하나 이상의 슬레이브 노드를 가질 수 있다.
- 자동 FailOVer: 마스터 노드에 장애가 발생하면, 나머지 노드들이 서로 통신하여 상태를 확인하고 해당 마스터의 슬레이브 중 하나를 새로운 마스터로 승격
- 데이터 일관성: Redis Clust는 최종적 일관성 모델을 따름
## 장점
- 수평적 확장성: 데이터가 늘어나면 노드를 추가하고 슬롯을 재분배하여 성능과 용량을 늘릴 수 있음
- 가용성: 일부 마스터 노드가 죽어도 슬레이브가 대신 역할을 수행함
- 고성능: 중앙 프록시 서버를 거치지 않고 노드와 직접 통신하므로 응답 속도가 빠름
## 단점
- 운영 복잡도
- 멀티 키 연산 제한: 여러 개의 키를 한꺼번에 처리하는 연산(MSET, SUNION)은 해당 키들이 같은 해시 슬롯에 있을 때만 가능
- 다른 노드끼리의 트랜잭션 불가능: MULTI 연산자를 사용하는 트랜잭션은 같은 NODE에서만 실행할 수 있어 트랜잭션 적용 불가
- 최소 노드 요구량: 고가용성을 보장하기 위해 최소 3대의 마스터 노드 필요

## Gossip Protocol
모든 노드가 모든 노드와 통신하지 않아도 되는 프로토콜
### 동작 원리
1. 각 노드는 정해진 시간 간격마다 클러스터 내의 무작위 노드 몇 개를 선택
2. 선택한 노드들과 자신이 알고 있는 정보를 주고 받는다
	1. 정보에는 자신과 다른 노드들의 상태, 슬롯 정보 등 포함
3. 정보를 받은 노드들 역시 다음 주기에서 또 다른 무작위 노드들에게 소문을 퍼뜨림
	1. 이 과정에서 정보가 클러스터 전체로 지수적으로 퍼져나감
### 주요 특징
- 확장성: 노드가 늘어나도 각 노드가 통신하는 대상은 항상 일정하여 네트워크 부하가 늘어나지 않음
- 결함 허용: 중앙 관리 서버가 없는 탈중앙화 구조로, 일부 노드가 죽거나 네트워크 패킷이 

## Sentinel과의 차이
- Sentinel은 데이터 분산 저장은 수행하지 않고 모든 노드가 동일 데이터를 보유함
