특정 노드에 장애가 발생하더라도 서비스를 지속할 수 있는 고가용성을 제공하기 위해 설계
### Redis Cluster 특징
- full-mesh 구조로 통신
- cluster bus라는 추가 채널(port) 사용
- gossip protocol 사용
- hash slot을 사용한 키 관리
- multi key 명령어가 제한됨
- 클라이언트는 모든 노드에 접속

## 해시 슬롯
- Redis Cluster는 데이터를 분산하기 위해 전체 데이터를 16384개의 논리적인 슬롯으로 나눔
	- 16384개인 이유: 서로의 상태를 확인하는 비트맵 크기가 작아지며, 노드가 1000대일 때 노드당 16개의 슬롯을 가질 수 있어 분산 처리에 충분함
- 클러스터에 참여하는 마스터 노드들은 이 슬롯을 나누어 가진다.
- key가 저장될 위치는 CRC16(key) (mod 16384) 공식을 통해 결정됨
	- CRC16을 쓰는 이유: 비트 연산위주라 빠름, key가 조금만 달라져도 결과값이 크게 변하여 분산에 용이
	- CRC16을 쓰면 키가 무작위로 분산되지만, 특정 키들을 반드시 같은 노드에 두고 싶다면, 중괄호를 이용한 Hash Tag를 사용하면 두 키가 항상 동일한 슬롯에 할당될 수 있음
## 고가용성과 복제
- Cluster 내의 각 마스터 노드는 하나 이상의 슬레이브 노드를 가질 수 있다.
- 자동 FailOVer: 마스터 노드에 장애가 발생하면, 나머지 노드들이 서로 통신하여 상태를 확인하고 해당 마스터의 슬레이브 중 하나를 새로운 마스터로 승격
- 데이터 일관성: Redis Clust는 최종적 일관성 모델을 따름
## 장점
- 수평적 확장성: 데이터가 늘어나면 노드를 추가하고 슬롯을 재분배하여 성능과 용량을 늘릴 수 있음
- 가용성: 일부 마스터 노드가 죽어도 슬레이브가 대신 역할을 수행함
- 고성능: 중앙 프록시 서버를 거치지 않고 노드와 직접 통신하므로 응답 속도가 빠름
## 단점
- 운영 복잡도
- 멀티 키 연산 제한: 여러 개의 키를 한꺼번에 처리하는 연산(MSET, SUNION)은 해당 키들이 같은 해시 슬롯에 있을 때만 가능
- 다른 노드끼리의 트랜잭션 불가능: MULTI 연산자를 사용하는 트랜잭션은 같은 NODE에서만 실행할 수 있어 트랜잭션 적용 불가
- 최소 노드 요구량: 고가용성을 보장하기 위해 최소 3대의 마스터 노드 필요

## Gossip Protocol
모든 노드가 모든 노드와 통신하지 않아도 되는 프로토콜
### 동작 원리
1. 각 노드는 정해진 시간 간격마다 클러스터 내의 무작위 노드 몇 개를 선택
2. 선택한 노드들과 자신이 알고 있는 정보를 주고 받는다
	1. 정보에는 자신과 다른 노드들의 상태, 슬롯 정보 등 포함
3. 정보를 받은 노드들 역시 다음 주기에서 또 다른 무작위 노드들에게 소문을 퍼뜨림
	1. 이 과정에서 정보가 클러스터 전체로 지수적으로 퍼져나감
### 주요 특징
- 확장성: 노드가 늘어나도 각 노드가 통신하는 대상은 항상 일정하여 네트워크 부하가 늘어나지 않음
- 결함 허용: 중앙 관리 서버가 없는 탈중앙화 구조로, 일부 노드가 죽거나 네트워크 패킷이 유실되어도 다른 노드를 통해 정보가 전달됨
### 장애 감지
- 노드 A가 노드 B에 Ping을 보냈는데 일정 시간 응답이 없으면 노드 A는 노드 B를 장애 의심 상태로 표시
- 이 정보가 가십 프로토콜을 통해 다른 노드들에게 퍼짐
- 과반수 이상의 노드가 노드 B를 장애 의심 상태로 동의하면, 노드 B는 공식적인 장애상태가 되고 Failover 시작

M노드가 ‘최초’로 FAIL판정되는 과정을 N노드 관점에서 이해해보자.
1. N은 과반수 이상의 다른 노드로부터 M이 PFAIL플래그로 설정된 메세지를 받았다.
2. N노드 역시 M이 PFAIL이라고 판단하고 있다.
위 두 조건을 모두 만족하면, N은 이제 M이 완전히 FAIL이라고 확정한다.

그리고 M이 FAIL이라는 정보를 Gossip section에 담아 다른 노드들에게 전파한다.
PFAIL의 전파와 다르게 FAIL상태의 전파는 ‘장애 확정'플래그이므로, 수신자 노드들은 수신 후 바로 M을 FAIL이라고 생각한다.

가십 프로토콜을 통해 분산된 노드들이 어떻게 하나의 결론(FAIL)에 도달하고, 슬레이브가 어떤 근거로 승격을 주장하는지 더 구체적인 데이터 구조와 예시를 들어 설명해 드리겠습니다.

## 1. 노드 상태 테이블(Node State Table)의 구성

각 노드는 자신을 포함한 클러스터 내 모든 노드의 정보를 담은 메모리 테이블을 유지합니다. 이 테이블에는 단순히 "살았다/죽었다"뿐만 아니라, "누가 누구를 죽었다고 생각하는지"에 대한 정보가 포함됩니다.

### 노드 A가 가진 상태 테이블 예시

| **노드 ID**  | **역할** | **상태**    | **의심 리포트 (Failure Reports)**                 |
| ---------- | ------ | --------- | -------------------------------------------- |
| **Node A** | Master | OK        | -                                            |
| **Node B** | Master | **PFAIL** | **[보고자: A, 시간: 14:01], [보고자: C, 시간: 14:02]** |
| **Node C** | Master | OK        | -                                            |
| **Node D** | Master | OK        | -                                            |
### 과반수 확인 과정 (메커니즘)
1. 개별 감지: 노드 A가 B와 통신이 안 되면, 자기 테이블의 B 항목에 "나(A)는 B가 의심스러움"이라고 리포트를 적습니다.
2. 소문 전파 (Gossip): 노드 A가 노드 C와 통신할 때, "나는 B가 PFAIL이라고 생각해"라는 정보를 패킷에 담아 보냅니다.
3. 정보 업데이트: 노드 C도 B와 통신이 안 되고 있었다면, C의 테이블에도 [A의 리포트]와 [C 자신의 리포트]가 쌓입니다.
4. 판단: 노드 A는 가십을 주고받으며 자기 테이블의 `의심 리포트` 목록을 채웁니다. 이때 리포트를 보낸 마스터의 수가 전체 마스터 과반수(예: 4대 중 3대)를 넘으면, 노드 A는 B의 상태를 `PFAIL`에서 `FAIL`로 확정하고 전파합니다.

## Sentinel과의 차이
- Sentinel은 데이터 분산 저장은 수행하지 않고 모든 노드가 동일 데이터를 보유함
