## RFP와 개발계약
### 제안요청서(RFP, Request for Proposal)
클라이언트가 개발사에게 제공하는 것으로 클라이언트가 희망하는 개략적인 요구사항을 개발용역을 수행하는 기업 혹은 조직에게 전달 
- 프로젝트에 대한 자세한 정보, 추진일정, 예산, 제안서 목차, 평가기준 등의 정확한 명시 필요함. 이는 제안서 품질 향상과, 클라이언트와 개발사 간의 의견 충돌 방지 가능 
- RFP 전달시 RFQ(Request For Quotation, 견적요청서)를 같이 전달하기도 함 
### 제안서(Proposal) 
RFP를 참고로 하여 프로젝트에 대한 제안서를 작성 
- 개발범위 및 내용, 인력투입계획, 일정계획 
- 사이트의 방향성 및 주요 컨셉트 
- 개발과정, 예산소요계획(견적서) 등의 내용 
### 개발사 선정 및 개발계약서 작성
제안서를 접수한 클라이언트는 제출된 제안서의 평가 및 검토 작업을 실시한 후에 어떤 업체에 프로젝트를 맡길 것인가 의사결정 
- 프로젝트 업체 선정 이후 개발계약서 체결 
- 계약금액, 개발범위, 하자보수, 납기관련 조항 
- 개발범위, 인력투입계획, 예산소요 계획 등

### 시스템 구축(Building the system) 
#### 아웃소싱
사이트 구축과 관련된 서비스를 제공하는 전문업체에 외주
#### 자체구축 vs 아웃소싱
자체 구축은 다양한 기술을 가진 팀 구성과 소프트웨어 도구 선택 필요 
- 자체 구축 
	- 장점 : 원하는 사이트 구축 용이, 비즈니스 환경 변화에 빠르게 대응 가능 
	- 단점 : 구축을 위한 학습과정이 오래 걸리며, 시장 진입이 늦어질 수 있음 
- 구축 패키지 활용 
	- 장점 : 자체 개발보다는 빠른 시장 진입 가능 
	- 단점 : 구축 패키지 비교 평가에 많은 시간 소요, 비즈니스 성격에 맞도록 패키지 수정시 비용이 많이 추가될 수 있음 
- 템플릿에 의한 구축 
	- 장점 : 가장 빠른 사이트 구축으로 빠른 시장 진입 가능 
	- 단점 : 템플릿에서 제공되는 기능에만 의존, 제한된 기능으로 사이트 구축과 운영

#### 자체 호스팅 vs. 코로케이션 vs. 아웃소싱(호스팅) 
- 호스팅(Hosting): 
	- 웹 서버의 운영관리와 24시간 상시 접속 서비스, 월/년별 사용료 지불 
	- 웹 서버 구축과 네트워크 접속 유지를 위한 기술과 인력 확보 부담 감소 
- 코로케이션(Co-location): 
	- 기업이 웹서버를 구매하거나 임대를 하여 그 운영에 통제권을 가지지만, 웹서버는 외부 관리 기업의 시설에 위치해 놓고, 관리 기업이 서버 시설과 통신회선을 유지보수 함 
- 아웃소싱(Outsourcing): 
	- 시스템 전체 또는 일부를 외부 전문 서비스 제공자에게 시스템 및 네트워크 관리, 소프트웨어 개발, 데이터 처리 등의 업무를 일괄 위탁 
	- 아웃소싱 기업이 동반 성장할 가능성이 있는지 파악 필요 
	- 사이트 백업과 유지보수 활동 및 보안 관리 모니터링 필요

자체 호스팅 => 모든 것을 스스로 관리 (내 집에서 직접 요리)
코로케이션 => 하드웨어를 소유하고 외부 시설을 이용 (외부 레스토랑에서 요리)
아웃 소싱 => 외부 제공업체에게 인프라와 서비스 위임 (외식 레스토랑에서 요리 주문)
### 테스트(Testing) 
#### 단위 테스팅(Unit Testing) 
프로그램 모듈을 하나씩 테스트 
- 통합 테스팅(System Testing) 
	- 사용자가 사이트를 사용할 때와 같은 방식으로, 사이트를 전체적으로 시험 
- 인수 테스팅(Acceptance Testing) 
	- 시스템이 원래 설계된 대로 실제 작동하는지 확인 
### 실행/운영(Implementation)
- 유지보수 비용: 개발 비용과 유사 
	- 전자상거래 사이트는 항상 수정, 개선의 과정이 필요함 
	- 웹 프로그래머, 디자이너, 마케터로 구성된 웹 관리팀에 의한 고객 피드백 필요 
- 벤치마킹(Benchmarking) 
	- 응답 속도, 웹페이지 레이아웃 품질, 설계의 관점에서 경쟁사들과 비교/개선 필요 

# 소프트웨어와 하드웨어 선택
## 단일 vs. 다계층 웹사이트 구조
### 시스템 구조
특정 기능을 수행하기 위해 필요한 정보시스템의 소프트웨어, 기계 및 작업의 배치
### 2계층 구조
웹 서버가 웹페이지의 요청에 응답하고, 데이터베이스 서버는 백앤드 에디터 저장소를 제공하는 전자상거래 시스템 구조
### 다계층 구조
웹 서버가 특정 업무를 수행하는 일련의 애플리케이션 서버를 포함하는 중간 계층에 연결될 뿐만 아니라, 제품, 고객, 가격정보를 포함한 회사의 기존 시스템의 백앤드 계층에도 연결되는 전자상거래 시스템 구조

ex) 아파치 엔진 X 등이 있다.
## 사이트 관리 도구
ex) Google Analytics
## 동적 페이지 생성 도구
ex) jsp, php

콘텐츠 편집도구, 협업, 관리도구를 포함함
## 애플리케이션 서버
### 웹 애플리케이션 서버(WAS)
- 웹 사이트에서 요구하는 특정 비즈니스 기능을 제공하는 소프트웨어 
- 미들웨어의 한 종류 
	- 기존 회사 내부 시스템과 고객들을 연결 
- 단일 기능의 애플리케이션들은 전자상거래에 필요한 모든 기능을 포함하는 통합 소프트웨어 도구로 대체되고 있음
## 전자상거래 머천트 서버 소프트웨어
머천트 : 상인
### 온라인 판매를 위한 기본 기능을 제공하는 서버용 S/W 
#### 온라인 카탈로그(Online catalog) 
웹 사이트에서 구매가능한 상품 목록 
고객 맞춤형 온라인 카탈로그를 만들 수 있는 데이터베이스 기능 포함 
#### 장바구니(Shopping cart) 
결제 이전에 구매희망 목록을 저장하고 확인할 수 있으며, 필요에 따라 목록을 수정하고 결제 
#### 신용카드 처리(Credit card processing) 
일반적으로 장바구니와 함께 동작  구매자의 신용카드를 식별하고, 카드에 요금을 청구하고, 대금을 정산함 
국내: PG(Payment Gateway, 전자지불서비스)사, VAN(Value Added Network)
## 머천트 서버 소프트웨어 패키지
복잡한 고객 중심의 사이트를 구축할 수 있도록, 필요한 모든 기능을 가진 통합 환경을 제공하는 소프트웨어 패키지

#### 확장성(Scalability) 
수요에 따라 하드웨어 규모를 확장할 수 있는 역량 
#### 수직적 확장 
개별 구성요소의 처리능력을 향상시킴
##### 장점
관리할 장비가 작고 관리 용이
##### 단점
몇 대의 빠른 컴퓨터에 의존함으로써 한 대가 다운되면 전체 작업 수행 불가능
#### 수평적 확장
단일 프로세서를 가진 서버를 여러 대 추가하거나, 서버의 역할을 분산
##### 장점 
이중화가 가능하여 장애 대처 용이 
##### 단점 
장비의 규모가 증가하고 관리 복잡

# 기타 전자상거래 사이트 도구
## 웹 사이트 디자인 : 기본적인 비즈니스 고려사항
최소한 고객들이 필요한 것을 찾고, 구매할 수 있어야 함.

## 전자상거래 사이트의 중요 디자인 고려사항 8가지
- 기능성
- 정보성
- 사용 편의성
- 중복적인 내비게이션
- 구매 편의성
- 다중 브라우저 기능성
- 간단한 그래핑
- 가독성
## 웹 사이트 최적화를 위한 도구
### 검색엔진 최적화 방법
- 메타태그(\<meta>), 타이틀, 페이지 콘텐츠, 키워드  
	- 검색 에이전트가 식별하는 키워드와 페이지 특징 설명 
- 틈새 시장 탐색 
	- 특정 집단을 끌어들일 수 있는 마케팅 키워드 결정 
	- ex) “보석” -> “1950년 보석＂ 
- 전문지식 제공 
	- 고객 유도와 신뢰 부여 
	- ex)분석자료, 가이드, 트랜드, 이슈, Q&A, FAQ 등 
- 링크 활용 
	- 다른 사이트와 URL 공유 
	- 블로그, 카페, SNS 
- 광고 구매 
	- 예산 범위내에서 검색 사이트에 직접 광고 노출 및 통계 효과 분석
- 지역 전자상거래 고려 
	- 지역 제품/서비스와 관련 있을 경우 지역명 키워드 고려
## 상호작용 및 액티브 콘텐츠를 위한 도구
- CGI (Common Gateway Interface) 
- ASP (Active Server Pages) 
- Java 
- JSP
- JavaScript
- Web 2.0 디자인 요소
	- 위젯
	- 메시업
		- 인터넷에서 제공되는 서로 다른 다양한 서비스를 합쳐서 새로운 서비스를 만드는 것
